# Standard rebaseline - 2026-07-20

This is a controlled before/after comparison on the same otherwise-idle local
environment:

- Filesystem: WSL2 `ext2/ext3` persistent volume (not tmpfs).
- Machine: Intel Core i5-7200U, 4 logical CPUs.
- Kernel: Linux 6.6.87.2-microsoft-standard-WSL2, x86_64.
- Toolchain: Rust 1.96.0.
- Profile: `standard`, five independent samples, 128-byte values.
- Current code: working tree based on `882c6c4`, including detailed worker
  timing instrumentation from this change.

Raw captures:

- `standard-2026-07-20-before-worker-cache.txt`: storage at `d008983` (before
  worker/group commit and application caches), with runner-only normalization
  to 200 durable TCP writes, randomized reads, warmed connections, and corrected
  barrier timing.
- `standard-2026-07-20-before-compaction.txt`: storage at `63c851b` (before
  streaming/background compaction); only the standard compaction scenario was
  selected.
- `standard-2026-07-20-after.txt`: complete current standard profile with cache,
  worker, WAL timing, and compaction attribution counters.

Runner-only changes select comparable scenarios or normalize workload setup;
historical storage code is unchanged. Values below are medians across all five
samples, never the best sample.

## Worker and group commit

| Scenario | Before | After | Before p99 | After p99 |
|---|---:|---:|---:|---:|
| Durable TCP PUT, c1 | 7 req/s | 7 req/s | 891 ms | 780 ms |
| Durable TCP PUT, c8 | 6 req/s | 76 req/s | 7.26 s | 587 ms |

At c8, every current sample committed 200 logical writes in 25 physical groups
with maximum group size eight. Median-sample fsync time was about 2.57 s total;
at c1, 200 groups spent about 28.5 s in fsync. WAL encoding and flush time were
negligible beside `sync_data`. This attributes the c8 gain to shared fsyncs,
while unchanged c1 throughput is the expected control.

The worker handoff still has a material read cost:

| Scenario | Before | After | Before p99 | After p99 |
|---|---:|---:|---:|---:|
| TCP GET, c1 | 9.86k req/s | 2.35k req/s | 512 us | 2.29 ms |
| TCP GET, c8 | 18.9k req/s | 10.8k req/s | 2.66 ms | 5.94 ms |
| TCP GET, c32 | 19.3k req/s | 13.9k req/s | 7.50 ms | 14.1 ms |

The detailed counters show no queue saturation, but expose queue/thread handoff
time on every read. A read fast path or separate read ownership is therefore a
measured follow-up, not a speculative optimization.

## SSTable cache

| Scenario | Throughput | p99 | Cache evidence per sample |
|---|---:|---:|---|
| Before application cache | 48.3k reads/s | 103 us | no application cache |
| Current cache disabled | 45.2k reads/s | 99.9 us | 101k file/block misses |
| Current 64 MiB cache | 301k reads/s | 17.0 us | 99,437 block hits / 1,563 misses |

The explicit disabled path matches the historical result within normal run
variation. Enabling the cache improves median throughput about 6.2x and p99
about 6.0x while retaining 1,563 decoded blocks / 22.4 MiB, below its 64 MiB
budget.

## Background compaction

| Scenario | Before | After |
|---|---:|---:|
| 300k-version end-to-end p50 | 3.24 s | 3.53 s |
| Median throughput | 92.6k versions/s | 84.9k versions/s |
| Foreground GET during merge | unavailable | 456k reads/s, p99 55.8 us |
| Peak reported merge buffer | all tables materialized | 2,032 bytes |

The after end-to-end measurement includes 50k concurrent foreground GETs and
manifest publication, so it is intentionally stricter than the old blocking
measurement. The median background build itself was about 3.04 s. Total
compaction time did not improve, but the request path no longer performs the
merge and its transient record buffer no longer scales with database contents.
All samples preserved 300k input/output versions and reduced about 50.1 MiB of
input tables to 43.6 MiB without foreground backpressure stalls.

## Conclusion

The standard profile confirms that group commit and the SSTable cache improve
the intended bottlenecks with attribution counters, and that background
compaction trades a small total-time cost for bounded merge buffers and
foreground availability. It also establishes TCP GET handoff as the remaining
Milestone 0 performance regression.
